Getting Started with Git

Introduction

Git is a version control system that tracks changes in code, enabling developers to manage versions, roll back to previous states, and collaborate efficiently. In courses like ELEC50014, Git facilitates code sharing among project partners (if this is part of the course) and further allows instructors to distribute project starter code to students.

Step 1. Create a GitHub account and install git

There are two components involved in setting up Git. One component is that you must have an account on a repository hosting service (that is, a website where you can put your code so your partner can access it). The repository hosting service we use in ELEC50014 is called GitHub. The other component is the Git command line tools, which allow you to interface with Git locally.

Step 1a. Create a GitHub account (if you don't have one already)

To sign up for a GitHub account, open GitHub.com in a browser and choose your username, imperial email, and password. Then, select the “free” plan, and then you will be taken to your homepage. Now, you need to verify your email account. Click on the tool icon in the top right corner to get to account settings, then select emails, and click the “verify” button next to your email address. Then, go to your email to confirm.C

Check if git is installed:

git --version

If you see a version number, git is already installed.

Step 1b. Install git (if not already installed):

On Linux (Ubuntu / Debian): 

sudo apt install git

On macOS (you first need to install Homebrew if you don't have it already): 

brew install git

Step 2. Configure Git with GitHub

We now need to configure Git to work with our GitHub account.

Step 2a. Generate and use a Personal Access Token (PAT)

We can use either the SSH or HTTPS protocols to connect Git from our local machine to the remote GitHub server. Both protocols provide a secure connection. We will use HTTPS, primarily for simplicity. To authenticate over HTTPS, we need a Personal Access Token (PAT), a secure alphanumeric code generated from GitHub, used instead of a password for Git operations over HTTPS.

Generate a PAT by following these steps:

  • Visit GitHub Personal Access Tokens.
  • Click "Generate new token (classic)".
  • Select Scopes: Choose repo for full access to repositories.
  • Click Generate Token and save the token securely (you won’t see it again).

Step 2b. Configure git with the PAT

Run the following commands to configure your Git account details:

git config --global user.name "Your GitHub Username"
git config --global user.email "your_github_email@example.com"

The first command sets your name. Remember to put it in quotes! The second sets your email. For this, use the email you used to create your Github account.

Authenticate with GitHub (on first push):

git push origin main

When prompted for a username, enter your GitHub username.

When prompted for a password, enter your Personal Access Token (PAT).

Cache your credentials to avoid repeated authentication:

git config --global credential.helper cache

Step 3. Create a repository

Create a new directory for the repository:

mkdir hello-git
cd hello-git

Initialize the git repository:

The following will create a repository in your current directory. Therefore, make sure to navigate to the place on your computer where you wish to store the repository (using cd etc. and then run the following commands.

git init

git init initializes a local repository, allowing git to track changes made in the folder.

Step 4 (this is optional if you have just created your first repository). Clone an existing repository (for example if created through the web interface)

For now, you may not need this step as you are creating your first repository. However, at some point you may need to clone a repository provided to you, or that you have created remotely on GitHub, onto your local machine.

Clone an existing repository:

git clone https://github.com/yourusername/repository-name.git

Note: GitHub provides a URL for every repository, ending with .git. On GitHub you can find this URL by going to your repository page and clicking on the Code drop down menu.

Navigate into the cloned repository:

cd repository-name

Overall, <git clone creates a copy of a remote repository on your local machine, making it ready for development.

Step 5. Open a local repository in VSCode

Open the repository folder in VSCode:

Make sure you have navigated into the repository folder. Then execute:

code .

This can also be done from within VSCode by selecting File > Open Folder... (choosing your repository folder)

From here on, you can open the shell from within VSCode by selecting Terminal > New Terminal...

This sets us up to use git from within VSCode, making code implementation and integration a seamless process.

Verify git integration:

  • VSCode automatically detects the repository if it’s initialized.
  • Open the Source Control (Explorer) tab on the left sidebar.

Step 6. Example: Managing a C++ File

Create a file: helloworld.cpp in VSCode (which is opened in your repository folder)

Stage the file:

git add helloworld.cpp

git add moves changes to the staging area, preparing them for a commit.

Commit the file:

git commit -m "Added helloworld.cpp"

git commit saves changes permanently to your local repository, along with a message describing what was done.

Modify the file: change Hello, World! to Hello, Git!

Stage and commit again:

git add helloworld.cpp
git commit -m "Updated message in helloworld.cpp"

Push changes to remote repository:

git push origin main

git push uploads committed changes from the local repository to a remote repository.

Step 7. Pull changes

Pull latest changes from remote repository:

git pull origin main

git pull downloads and integrates updates from the remote repository into your local project.

Basic Git Commands

This is a summary of the basic Git commands you have seen above.

A quick reference to some of the most common git commands is available here.

  • git status - Tells you the status of your local repository: what files have been added, deleted, and changed since your last commit. It also shows which changes are in staging and which ones aren’t. When a change is in staging, it will be included in the next commit. If you have text coloring enabled, staged changes appear green and unstaged changes appear red (the same colors appear in VSCode).
  • git add <file/folder> - Stages the specified file or folder. You can use regular expressions to include multiple files or run git add multiple times for different files. If you stage a file but make changes afterward, it automatically gets unstaged, so you must run git add again.
  • git commit - Commits the changes that were staged. Think of this as logging a new version of your code with the set of staged changes. Git will open its text editor so that you can add a message describing the changes. Writing clear and informative commit messages can help your collaborators (and your future self) understand what was changed in previous commits. Note that all commits only exist on your local copy until you push them to the remote repository.
  • git push - Pushes all local commits since your last push to the remote repository. This allows collaborators to access your changes. If someone else has made changes to the remote repository while you were working, Git may require you to perform a git pull before you can push your changes.
  • git pull - Pulls all commits from the remote repository that your local version doesn’t have. This retrieves any changes that others working on your project have pushed since the last time you pulled. You should commit all your changes before doing this to avoid losing them.
